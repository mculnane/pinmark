# CLAUDE.md

Guidance for Claude Code working in this repository.

## What this is

Pinmark ‚Äî a Safari Web Extension (Manifest V3) that saves the current tab to
Pinboard with frequency-ranked tag autocomplete. Public repo:
<https://github.com/mculnane/pinmark>. Personal/local use; not on the App Store.

## Layout

- `src/` ‚Äî the web extension, and the **source of truth**. Edit here.
  - `manifest.json` ‚Äî MV3 manifest
  - `popup/` ‚Äî popover UI; `popup.html` loads `popup.js` as an ES module
  - `background/` ‚Äî service worker (classic script ‚Äî see gotchas)
  - `lib/` ‚Äî `browser.js` (cross-browser API shim), `storage.js`,
    `pinboard-api.js` (transport-swappable client), `tag-autocomplete.js`
    (pure ranking/tokenizing logic)
  - `icons/` ‚Äî generated PNGs
- `xcode/` ‚Äî Xcode container app + extension target, generated by
  `scripts/convert.sh`. It **references** `src/` via relative paths (no
  `--copy-resources`), so editing `src/` and rebuilding picks up the changes.
- `tests/` ‚Äî `*.test.mjs`, runnable without Node (see below)
- `scripts/` ‚Äî `convert.sh` (regenerate the Xcode project), `generate_icons.py`

## Build & run

1. `open xcode/Pinmark/Pinmark.xcodeproj` (run `./scripts/convert.sh` first if
   `xcode/` is missing or the manifest structure changed).
2. Set the signing Team on both targets (`Pinmark`, `Pinmark Extension`) with
   automatic signing.
3. ‚åòR, then enable in Safari ‚Üí Settings ‚Üí Extensions.

**Sign with a real team, not Team: None.** With a personal Apple ID team selected
(step 2) the extension is signed with a development cert and **persists across
Safari restarts** (until the free cert expires ~7 days later). An *unsigned* build
(Team: None / ad-hoc) only loads while Safari ‚Üí Develop ‚Üí **Allow Unsigned
Extensions** is on ‚Äî and that toggle **resets on every Safari quit**, so the
extension silently disappears from Settings ‚Üí Extensions whenever Safari restarts.
This bit us hard during debugging; always select the team.

**Safari loads the extension from the built `.appex`, not from `src/` directly.**
Rebuild (‚åòR) for any JS/HTML/CSS change to take effect.

## Tests

There is no Node on the primary machine; tests run under JavaScriptCore:

```sh
JSC=/System/Library/Frameworks/JavaScriptCore.framework/Versions/A/Helpers/jsc
"$JSC" -m tests/tag-autocomplete.test.mjs   # autocomplete ranking/tokenizing
"$JSC" -m tests/pinboard-api.test.mjs       # API error classification
```

`jsc` lacks `URL`/`URLSearchParams`/`fetch` ‚Äî tests shim or inject these. To
syntax-check a module: `"$JSC" -m <file>` (a `SyntaxError` is a real problem; a
runtime `ReferenceError` for `document`/`browser` just means there's no DOM under
jsc, and the syntax is fine).

## Conventions & gotchas

- **The API layer is transport-swappable.** The UI only calls
  `createPinboardClient()` from `lib/pinboard-api.js`. To route through an MCP
  server later, add a transport with the same `request(path, params)` method ‚Äî
  don't touch the UI.
- **`pinboard-api.js` is the only module that knows the transport.** Its errors
  carry a stable `code` (`unauthorized` / `rate_limited` / `server_error` /
  `network`); the popup switches on `code`, never on message text.
- **The background service worker must stay self-contained** ‚Äî no ES `import`.
  Safari's support for module service workers is unreliable, so
  `background/background.js` duplicates the few storage keys it needs. Keep them
  in sync with `lib/storage.js`.
- **Network requests go through the background worker, not the popover.** On
  Safari a cross-origin `fetch` from the popover is CORS-blocked even with the
  host granted (Pinboard sends no CORS headers); only the background worker is
  exempt. So the popup's API client is given `lib/background-fetch.js` as its
  `fetchImpl`, which messages `background.js` (`type: "pinboard-fetch"`) to run
  the real `fetch` and returns a reconstructed Response. Keep the actual `fetch`
  in the background ‚Äî don't "simplify" it back into the popup.
- **Rate limit:** Pinboard asks for ‚â•3s between calls; the client serialises and
  spaces requests. Don't remove the limiter.
- **Token** lives only in `browser.storage.local`, entered via the popup's
  settings panel. Never read it from disk, never commit it.
- **Signing / `project.pbxproj`:** the committed `DEVELOPMENT_TEAM` is empty on
  purpose (this is a public repo). A local build re-adds your own Team ID, but the
  file is marked `skip-worktree` so it won't be staged. Don't commit a real Team
  ID. To make a genuine project-structure change, first run
  `git update-index --no-skip-worktree xcode/Pinmark/Pinmark.xcodeproj/project.pbxproj`.
- **A free Apple ID signing certificate expires every 7 days** ‚Äî rebuild to
  refresh.

## Debugging in Safari (hard-won)

- **The popover has no "Inspect Element"** ‚Äî right-clicking it only offers
  *Reload*, so you can't open Web Inspector on the popup the usual way. To debug
  popup logic, temporarily surface the underlying error in the on-screen message
  (e.g. append `err.cause?.message` in `describeError`), rebuild, and read it. For
  the background worker's console, use Safari ‚Üí Develop ‚Üí **Web Extension
  Background Content ‚Üí Pinmark**.
- **The background service worker is sticky.** ‚åòR reloads the popup/HTML/JS, but
  Safari often keeps the *old* worker running, so changes to `background.js` may
  not take effect. Toggle Pinmark off/on in Settings ‚Üí Extensions to force a fresh
  worker. (Quitting Safari also works but disables unsigned extensions ‚Äî see Build
  & run ‚Äî so only do that when signed with a team.)
- **"Couldn't reach Pinboard" / a `network` error with the host already granted**
  is the popover-fetch CORS issue, not a permission problem. The fix is background
  routing (see gotchas); more permission fiddling won't help.
- **Granting `api.pinboard.in` (Settings ‚Üí Websites) is necessary but not
  sufficient** ‚Äî the request still has to originate from the background worker.
- **Check Pinboard's API independently of the extension** before assuming a local
  bug:
  `curl -s -o /dev/null -w "%{http_code}\n" "https://api.pinboard.in/v1/tags/get?auth_token=dummy:0000&format=json"`
  ‚Äî `401` = healthy (a bad token is rejected), `5xx` = Pinboard outage, timeout =
  unreachable. Pinboard ran an extended `500` outage during initial testing, which
  masked the real (CORS) bug for a while.
